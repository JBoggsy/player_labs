Mentalist v4 — Player-SDK Rewrite A stable, observable player built on the SDK's now-shipped game-agnostic helpers, with a classifier rebuilt for the axis-combo judge.

Status: Draft for review (rev 2) Date: 2026-06-15 Supersedes: mentalist v1–v3 Game: cue_n_woo · axis_combo SDK: P1–P5 on main · locked

Two forces converge on a rewrite. The game changed the ground we stand on — the judge is no longer one of 61 named styles but a random 4-tuple drawn from 15 axes, so the current 61-way classifier returns near-uniform noise. And we want a player we can keep: robust, observable, built on the same shared foundation as the other lab players. Since rev 1 of this doc, a focused SDK pass shipped the exact generalizations that make this clean (P1–P5): a protocol-agnostic websocket bridge, a shared LLM-client helper, a reusable trace-config, an opaque trace step label, and a guaranteed grid-free telemetry core — now merged to main (#67–#71) and locked into this lab. This revision builds the rewrite on those concrete, installed APIs and corrects one wrong assumption from rev 1.

What changed since rev 1

Rev 1 proposed (in its §9) six SDK generalizations and assumed we'd hand-roll the bridge and that adopting telemetry "pulls in mettagrid types." A subsequent SDK pass implemented P1–P5 and verified that the telemetry core was already grid-free — so the wrong assumption is removed, and §3/§7 below now build on real APIs (run_message_bridge, players.player_sdk.llm, TraceConfig, TraceEvent.step, players.player_sdk.telemetry) instead of sketches. Those PRs (#67–#71) are now merged to players main (tip fdf2987) and locked into this lab (uv.lock); all P1–P5 helpers import in the lab .venv and the lab suite (385 tests) is green on the bump. See the SDK-side report: players/player_sdk/docs/designs/generalizing-the-sdk-for-turn-based-games.md.

1Why rewrite now

Mentalist:v3 is the league champion but sits rank 3 of 3. The standing is real — Bedrock works in the tournament pods (verified from fresh episode logs), and the WEMA-over-last-20 standing already reflects the LLM player, not the old fallback era. The player is losing on the merits, for two reasons that no single knob can fix:

Root cause A · the classifier is blind

The judge now steers toward a combination like technical; evidence-first; frontier town; sterile — four values from four of 15 axes (~287M combinations). The 61-style TF-IDF nearest-neighbor has no valid target; on that concept it returned haiku 0.354 | urban-planning 0.347 | sermon 0.341, a three-way tie of unrelated styles. It solves a problem that no longer exists.

Root cause B · the writer is mis-aimed

The new judge rewards concrete, terse, on-topic answers. Opponents win with "A brass key" / "A train ticket" (judge p = 0.95–0.999) while mentalist writes evocative prose ("Threshold stone worn smooth by countless passages") and scores p = 0.00–0.17. The writer's whole prompt strategy is tuned for the old "play a named style" game.

Since we're replacing the two load-bearing components anyway, this is the moment to also put the player on a durable footing: structured tracing, a per-episode telemetry artifact, and the shared SDK conventions the rest of the lab uses. Hence a full rewrite — v4 — and the timing is now ideal because the SDK just grew the pieces a turn-based text player needs.

2What the SDK now offers (P1–P5)

The SDK is still a "two-loop cyborg-agent framework" whose center of gravity is tick-based gridworlds. That part is unchanged and we still won't use it (see below). What changed is that the SDK now exposes the game-agnostic helpers a turn-based player actually wants, as additive, tested, backward-compatible modules.

The runtime layer (still does not fit — and that's fine)

AgentRuntime.step(), Mode/Reflex/StrategyRunner model a per-tick perceive→belief→mode→action loop for continuous gridworlds. Cue-n-Woo is turn-based custom JSON (state → ask/propose/answer), 4 phases, ~7 actions per episode. There is no every-frame reflex to model, so we deliberately do not adopt the tick runtime. The SDK report agrees and explicitly keeps the grid machinery as an opt-in module beside a game-agnostic core.

3Adoption strategy

Build the player as a thin handler over run_message_bridge, wrapped in TraceOutputs, with a pure phase state machine doing the thinking. The bridge owns transport and clean shutdown; our code owns decisions and is transport-free and unit-testable.

Sketch of the entry point — note how little transport code we now own:

from players.player_sdk import run_message_bridge, TraceOutputs, TraceConfig

async def main() -> None:
    cfg = TraceConfig.from_env(env_prefix="MENTALIST", groups=EVENT_GROUPS)
    with TraceOutputs.from_env(prefix="MENTALIST", event_filter=cfg.allows) as out:
        engine = PhaseEngine(trace=out.trace_sink, metrics=out.metrics_sink)
        await run_message_bridge(
            os.environ["COWORLD_PLAYER_WS_URL"],
            handler=engine.handle,        # (frame) -> list[str] outbound frames
            trace_outputs=out,            # bridge guarantees out.close() on teardown
            ping_interval=None, max_size=None,
        )
    # exit-0-on-unclean-close is the bridge default; no bespoke close handling.
Why this is the right line

It takes the SDK's real, now-tested game-agnostic value (transport + the exit-0 rule, telemetry, artifact upload, LLM plumbing, trace config) and skips the tick runtime that doesn't fit. We own only what's genuinely game-specific: the phase logic, the classifier, and the prompts. And by being the second consumer of these helpers, mentalist v4 is the forcing function that proves them (§9).

5The new axis classifier

select_axis_combo_concept picks 4 distinct axes of 15, then one value from each, joined with "; ". The space is huge (~287M) but factored — only ~326 atomic values exist. So the tractable question is not "which of 287M" but "for each axis, which value (if any) is the judge exhibiting?"

The 15 axes

syntax16domain30emotion12

Inference design (proposed)

  1. Probe across axes. Re-select the 3 private questions so the judge's axis values surface in its answers (a frontier-town/sterile/technical judge reveals all four under good open prompts).
  2. Score each axis independently against per-axis reference fingerprints built offline from worker draws; emit a ranked (value, score) list per axis.
  3. Don't force 4. We don't know which 4 axes are active, so emit a confidence per axis and let the writer consume the most confident handful. Concrete axes (object, place, domain) are both easiest to detect and highest-leverage to echo (§6), so they get priority when present.
  4. Stay LLM-free and cheap (the probe-validated virtue): TF-IDF/char-ngram cosine over known strings, deterministic, no runtime network.
Open empirical question · re-probe IN FLIGHT

Axis separability from 3 answers is being measured now against the live cue-n-woo-fleet.softmax-research.net worker (probe/probe_axis_recovery.py): for 5 axes (the 3 concrete bets — object, place, domain — plus register/rhetoric as an abstract contrast), it builds single-axis reference fingerprints and tests recovery of the true value from realistic 4-axis blended draws, reporting per-axis top-1/top-3 for word_raw and char_3_5. The old 95% top-1 was the 61-style game and does not transfer; this number gates the classifier design. If some axes prove near-undetectable, the writer leans on the transcript for those, exactly as the confident-axis design already allows. Results will land in docs/probe-findings.md.

6Writer & strategy retune

Keep the retry policy, tool-forced output, and graceful fallback; move the client construction to the SDK helper; change what we ask for:

  • Concrete over evocative. "A brass key" beats "Threshold stone worn smooth by countless passages" at p=0.95. Prompt the writer to name the concrete thing — especially the detected object/place/domain value — in the first one or two words, within the 12-token cap.
  • Feed axis values, not a style label. The prompt takes the top per-axis detections + the raw transcript, not one "your style is X" sentence.
  • Author axis-exploiting challenge questions (like the winners' "what image should the quiet doorway cue point to?") whose favored answer is a specific axis value a blind opponent can't guess. Replace today's generic open prompts (now low-signal / duplicate-conflict traps).
  • Keep the duplicate-conflict guard — still correct under unchanged scoring.
Honest API nuance · the LLM helper

The SDK's call_json is built for text responses parsed caller-side (extract_json_object), whereas our writer relies on tool-forced output (tool_choice → submit_answers), which is stricter for the ≤12-token answer contract. So mentalist adopts the helper's client/model/usage pieces (select_client, resolve_model, bedrock_enabled, usage_dict) but keeps its own tool-call path rather than call_json. That gap — "a call_tool variant for tool-forced output" — is logged as feedback to the SDK (§9), a concrete improvement nominated by being a real second consumer.

10Risks & open questions

  • SDK dependency. resolved — P1–P5 merged to main (#67–#71) and locked into the lab (fdf2987); helpers import and 385 tests pass. No longer a blocker.
  • Axis separability is unproven. The classifier rests on detecting axis values from 3 answers. Mitigated by re-probing first (§8.2) and by the confident-axis-only design that degrades to transcript-only when an axis is unreadable.
  • LLM-helper fit. The helper targets text/JSON, not tool-forced output; we adopt its client pieces and keep our tool path (§6). Low risk, but a real seam to feed back to the SDK.
  • haiku vs opus. The writer is on haiku-4-5 (the only model the tournament Bedrock role can invoke). Concrete/terse answers may need less model strength than prose did — possibly a tailwind; the --secret-env ANTHROPIC_API_KEY path remains a hedge.
  • Config drift. If the league changes concept_axis_count or the axis files, the per-axis references need regenerating — documented in data/axes/.
  • Scope. Large rewrite; the staged plan (§8) keeps each step independently verifiable and reversible, and no league submission happens until a matched eval beats v3.
