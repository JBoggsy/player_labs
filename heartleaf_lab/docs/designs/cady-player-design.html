Problem & goal§1

A player that plays — before a player that wins

Heartleaf has no policy of ours yet. Cady v1 exists to prove the whole loop end to end: connect to the /player socket, decode the Sprite-v1 scene, move around the map, collect vegetables, and be inside its own house at 6 PM so any gnome who wanders in becomes a scoring guest.

Scoring in Heartleaf is hosted food × guests, and only hosts score. The real lever is social — recruiting a full table over chat — but that is a deliberate v2. v1 is the clean baseline it builds on: it must genuinely gather and navigate, the two core competencies, and everything else is layered on from there.

Goals — v1

  • Connect, decode Sprite-v1 frames, exit 0 on close.
  • Navigate with a bang-bang movement controller.
  • Gather: find food gardens, path to them, press A.
  • Host: return home before dinner, hold inside at 6 PM.
  • Emit trace events + metrics via the SDK sinks.
  • Modular: perception, policy, action are separable units.

Non-goals — deferred

  • Chat / invitations to recruit guests — the reserved v2.
  • Host-vs-visit scheduling; visiting; multi-day food banking.
  • Any LLM decision layer.
  • A* routing (straight-line steering first).
  • Broad test coverage — the hosted eval is the test.

Foundation§2

The SDK now owns the Sprite-v1 transport

The Player SDK now includes players.player_sdk.sprite_bridge — the BitWorld /sprite_player transport bridge, the SpriteV1 peer of the cogweb and mettagrid bridges. Cady no longer vendors a wire layer. Everything below is imported from players.player_sdk:

Transport & decode — the SDK’s

run_sprite_bridge(url, decide, …) connects, accumulates the binary record stream into a raw SpriteWorld (objects + sprites with labels), calls decide per changed frame, packs the mask/chat, and exits 0 on close. Plus Button, env_ws_url, Trace/Metrics sinks.

The game — ours

Only three things: interpret the SpriteWorld into Heartleaf state (perception), decide what to do (policy), and turn that into a Button mask (action). The transport and the raw decode are no longer our code.

The bridge’s one callback is decide(world, ctx) → mask | (mask, chat) | None — return a mask to hold those buttons, add a chat line, or None to hold the last mask. It is robust by construction: undecodable frames and bad outputs are skipped, never raised, so the process survives the whole episode.

Prerequisite — bump the pinned SDK (shared with crewborg) Our pyproject.toml pin (6dcd022…) predates the bridge. Bump it to e8921a6… (current main, incl. PR #20) + uv lock, and mirror it in crewrift_lab/tools/versions.env. Because crewborg imports the same SDK, run its tests after the bump to confirm no regression — additive change, but verify. This is build step 1.

Per-tick data flow§3

One tick, end to end

The SDK bridge decodes each binary frame into a raw SpriteWorld and calls decide — that call is the boundary. Right of it is all Cady: resolve a Heartleaf view, update belief, pick a mode, produce a held-button command. The bridge sends the mask (only on change) and owns the exit-0-on-close contract.

Architecture§4

Layers, each with one job

The package heartleaf_lab/cady/cady/ — no wire/ layer, because the SDK owns the wire now. Perception sits at the bottom on the SDK’s SpriteWorld; the transport entry point is at the top. Colour marks ownership: what we write fresh and what the SDK supplies.

The cyborg contract§5

Six types make it a proper SDK policy

AgentRuntime is generic over six type parameters and three pure functions. Filling them for Heartleaf is what makes Cady a real cyborg policy rather than an ad-hoc loop — and it forces perception, belief, and action apart at clean seams. The bridge and the runtime meet in exactly one place: decide.py wraps the SDK’s SpriteWorld in an Observation, calls step(), and unpacks the returned Command into the bridge’s (mask, chat).

Deterministic policy§6

Gather all day, be home for dinner

v1’s brain is a clock, not a model. It mirrors the villagers’ time policy without an LLM: gather while the day is young, then stop and get home in time to host. The thresholds live in one config block so every iteration is attributable.

Action resolution

Cady reimplements crewborg’s bang-bang d-pad algorithm (not its code — the wire is the SDK’s now): steer toward a target world point with a release-near-target deadband and a predictive stop, emitting a Button mask. Intents map to it directly — navigate_to steers straight at a point; gather_at navigates to a garden then fires a fresh A press in range; enter_house navigates to the entrance then presses A to go inside; hold/idle emit the neutral mask.

Wire vocabulary§7

Everything v1 needs is in the labels

The bridge hands us world.objects joined to world.sprites[id].label — and Heartleaf’s protocol.nim + heartleaf.nim encode everything v1 needs in those labels and object positions. So v1 does no sprite-pixel decoding at all; talking_villager.nim is the reference for turning these into behaviour.

Testing§8

Minimal, high-value only

Per the lab’s discipline, the hosted eval is the real test. The few unit tests worth writing pin the load-bearing seams:

  • Perception — a synthetic SpriteWorld with a "garden marker" object and "clock 3"/"clock :"/… glyph objects → assert garden position, food flag, and parsed time. (The SDK owns + tests the frame decoder — we don’t re-test the wire.)
  • Policy — assert the clock threshold flips GatherMode → HostMode, and no-food falls back cleanly.

No coverage for its own sake — no smoke tests, no pre-upload gate.

Risks & open questions§9

What could bite, and the mitigation

  1. Self / seat identity — the stream doesn’t announce our seat and self isn’t a distinct object. Mitigation: crewborg’s answer is self = the camera centre (world-map object); the capture probe confirms it and which "gnome <i>" is ours before nav is trusted. First milestone.
  2. House / garden trigger geometry — where to stand and press A. Mitigation: mirror talking_villager.nim, confirm with the probe.
  3. SDK pin bump is shared with crewborg — bumping 6dcd022…→e8921a6… moves crewborg’s SDK too; run its tests after the bump. Additive change, but verify.
  4. Local run path — the bridge’s path constant is /sprite_player but the local server serves /player; the bridge uses the injected URL verbatim, so confirm the hosted COWORLD_PLAYER_WS_URL path for this game.
  5. Python-image upload path — the manifest’s player run is /bin/<name>; confirm the Python image uploads cleanly for this game (as crewborg does) before the first upload.
  6. League existence / game version — verify via the Observatory API before the first experience request. Carried over from lab creation.
  7. Pixel decode is deferred, not free — A* over the walkability sprite would need pixel/palette decode (the bridge exposes raw SpriteDef.data). v1 avoids it entirely by steering straight-line.

Build order§10

The path to a first eval

  1. Bump the SDK pin (6dcd022…→e8921a6…) in pyproject.toml + uv lock + versions.env; confirm run_sprite_bridge imports and run crewborg’s tests (shared-SDK check).
  2. Capture probe — a logging decide that dumps world labels + positions from a real stream; calibrate self/seat + garden/house geometry.
  3. perception.py + types.py — the six types, perceive (labels → state, clock-glyph join), update_belief.
  4. action.py — bang-bang nav + gather-press + enter-house, emitting Button masks.
  5. Modes + strategy (Gather → Host) + runtime.build_runtime + decide.py adapter.
  6. main.py (run_sprite_bridge + tracing), Dockerfile, README, VERSION_LOG.
  7. Build image → local smoke (connects / plays / exits) → upload → first hosted eval.
