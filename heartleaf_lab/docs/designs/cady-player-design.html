Problem & goal§1

A player that plays — before a player that wins

Heartleaf has no policy of ours yet. Cady v1 exists to prove the whole loop end to end: connect to the /player socket, decode the Sprite-v1 scene, move around the map, collect vegetables, and be inside its own house at 6 PM so any gnome who wanders in becomes a scoring guest.

Scoring in Heartleaf is hosted food × guests, and only hosts score. The real lever is social — recruiting a full table over chat — but that is a deliberate v2. v1 is the clean baseline it builds on: it must genuinely gather and navigate, the two core competencies, and everything else is layered on from there.

Goals — v1

  • Connect, decode Sprite-v1 frames, exit 0 on close.
  • Navigate with a bang-bang movement controller.
  • Gather: find food gardens, path to them, press A.
  • Host: return home before dinner, hold inside at 6 PM.
  • Emit trace events + metrics via the SDK sinks.
  • Modular: perception, policy, action are separable units.

Non-goals — deferred

  • Chat / invitations to recruit guests — the reserved v2.
  • Host-vs-visit scheduling; visiting; multi-day food banking.
  • Any LLM decision layer.
  • A* routing (straight-line steering first).
  • Broad test coverage — the hosted eval is the test.

Foundation§2

The SDK ships no Sprite-v1 — so vendor the wire, use the SDK for the rest

The Coworld Player SDK is engine-agnostic and contains no Sprite-v1 support; that binary substrate lives only in the crewborg example player. The split falls out cleanly:

Vendored · ported from crewborg

The Sprite-v1 byte decoder, the retained SceneState, and the input/chat encoders + button bits. This is the wire contract, not Crewrift semantics — Cady owns its copy, so it has no dependency on crewrift_lab.

Used from players.player_sdk

Transport — run_message_bridge with exit-0-on-close. Brain — AgentRuntime + Mode / ModeRegistry + SynchronousStrategyRunner. Observability — TraceSink / MetricsSink.

crewborg’s build_runtime is the assembly template; its decoder.py, scene.py, and action.py are the port sources. What we write fresh is everything Heartleaf: interpreting the scene and deciding what to do with it.

Per-tick data flow§3

One tick, end to end

Each binary frame folds into the scene, resolves into a Heartleaf view, updates long-lived belief, drives mode selection, and produces a held-button command — sent only when it changes. The loop runs continuously; the SDK bridge owns the socket and the exit-0-on-close contract.

Layers, each with one job

The package heartleaf_lab/cady/cady/ stacks from the wire contract at the bottom to the transport entry point at the top — mirroring how crewborg and mentalist vendor their players. Colour marks ownership: what we ported, what we write fresh, and what the SDK supplies.

The cyborg contract§5

Six types make it a proper SDK policy

AgentRuntime is generic over six type parameters and three pure functions. Filling them for Heartleaf is what makes Cady a real cyborg policy rather than an ad-hoc loop — and it forces perception, belief, and action apart at clean seams.

Deterministic policy§6

Gather all day, be home for dinner

v1’s brain is a clock, not a model. It mirrors the villagers’ time policy without an LLM: gather while the day is young, then stop and get home in time to host. The thresholds live in one config block so every iteration is attributable.

Action resolution

Cady ports crewborg’s bang-bang d-pad controller: steer toward a target world point with a release-near-target deadband and a predictive stop. Intents map to it directly — navigate_to steers straight at a point; gather_at navigates to a garden then fires a fresh A press in range; enter_house navigates to the entrance then presses A to go inside; hold/idle emit the neutral mask.

Not in the scheme — calibrate first The scheme gives the object vocabulary but not the geometry: the self-position camera offset (self is the camera centre plus a fixed offset) and the house/garden trigger distances. Testing§8

Minimal, high-value only

Per the lab’s discipline, the hosted eval is the real test. The few unit tests worth writing pin the load-bearing seams:

  • Decoder — hand-built Sprite-v1 frames (ported wire-test helpers) → assert the SceneState tables decode.
  • Perception — a synthetic scene with a "garden marker" and a "clock " sprite → assert garden position, food flag, parsed time.
  • Policy — assert the clock threshold flips GatherMode → HostMode, and no-food falls back cleanly.

No coverage for its own sake — no smoke tests, no pre-upload gate.

Risks & open questions§9

What could bite, and the mitigation

  1. Self-position offset — unknown exact value. Mitigation: the capture probe calibrates it before nav is trusted. First milestone.
  2. House / garden trigger geometry — where to stand and press A. Mitigation: mirror talking_villager.nim, confirm with the probe.
  3. Straight-line nav vs. A* — v1 steers straight; add A* over the streamed walkability grid only if an eval shows wedging.
  4. Python-image upload path — the manifest’s player run is /bin/<name>; confirm the Python image uploads cleanly for this game (as crewborg does) before the first upload.
  5. League existence / game version — verify via the Observatory API before the first experience request. Carried over from lab creation.

Build order§10

The path to a first eval

  1. Vendor + test the Sprite-v1 wire layer (wire/), with the capture probe.
  2. Calibrate the self-offset and garden/house geometry from a captured Heartleaf stream.
  3. perception.py + types.py — the six types, perceive, update_belief.
  4. action.py — navigate, gather-press, enter-house (ported controller).
  5. Modes + strategy (Gather → Host) + runtime.build_runtime.
  6. main.py (SDK bridge + tracing), Dockerfile, README, VERSION_LOG.
  7. Build image → local smoke (connects / plays / exits) → upload → first hosted eval.
